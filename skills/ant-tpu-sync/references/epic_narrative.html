三条线同时冲刺

5 月是全书高潮，三件事并行：

③ 方法论 · 如何系统性地把一个项目搬上 TPU 并优化

如果把这个项目抽象成一套可复用的「迁移 + 优化」方法论，它有 7 个阶段。 每个阶段都给出目标、关键动作、本项目里的真实数据，以及对应的证据文档。

1

可行性验证：先用一个已知模型探路

不要一上来就搬自家最复杂的模型。先拿一个结构相近的公开模型（这里是 Qwen3-Next）做端到端 smoke test， 同时建立 GPU 与 TPU 两侧的精度 + 性能基线，并用一次 profiling 找出第一个瓶颈。

  • 精度基线：forward 对齐 e-4、backward 对齐 e-3（jax-cpu vs torch-gpu vs jax-tpu 三方对比）。
  • 性能基线：TPU 64 芯片 4×4×4、完整 48 层、BS=2 / seq=8k，single step 6.58s（≈2489 tok/chip/s）。
  • 第一个瓶颈：linear attention 的 pure-JAX 实现反向激活显存过大 → recompute → batch 上不去。结论：从 kernel 入手，按 roofline 调并行。
📄 Qwen 测试📝 Benchmark 配置
2

看懂目标模型：结构先于代码

搬之前必须把目标模型的结构吃透，否则会「搬错」。Ling V3 的关键特征：KDA（取代 Ling2.5 的 GLA）+ 门控 MLA（逐层 hidden_state 决定激活哪些 attention head）+ Muon 优化器 + QK_Clip。 层结构：RMSNorm → Attention(KDA/MLA) → RMSNorm → MLP/MoE，配残差。Tiny 24 层、Flash 42 层（前 2 层 dense）。

  • 把模型按「哪些是标准 Transformer、哪些是自定义算子」分类——自定义的（KDA、门控 MLA）才是移植和精度对齐的重点。
  • 提前列出「需要移植的优化项」清单和框架层分工，避免后期返工。
📄 Ling V3 结构📄 工作分工📄 移植清单
3

精度对齐：逐算子对到参照实现

这是迁移里最枯燥也最关键的一步。以原训练框架（Megatron）为黄金参照，逐算子把数值对齐到 1e-4。 自定义算子（MLA、KDA）单独立项验证，因为它们最容易在 TPU/JAX 上出现数值偏差。

  • 对齐顺序：先 forward 后 backward；先单算子后整层后端到端。
  • 把对齐过程记成实验记录，每个偏差都要有根因，不能靠「调一调就过了」蒙混。
📄 V3 精度对齐📄 MLA 工作事项📄 实验记录
4

工程平台：让实验「可反复、可观测」

优化是一个高频迭代的过程，没有好平台就会被环境问题拖死。本项目用 GitOps：算法工程师只改 3 行 YAML、提 PR、merge 即 kubectl apply； 配套 MLDiagnostics SDK（loss/step_time/TFLOPS/MFU 等 8 类指标 + xprof profiling）和 Cloud Monitoring dashboard。

  • 关键约定：JobSet + hostNetwork、tpu7x-standard-4t node pool、libtpu 必须钉死在 0.0.36（0.0.35/0.0.37 有 HLO regression）。
  • 埋点纪律：MLRun 调用必须 try/except 包裹，否则监控故障会拖垮整个训练 Job。
  • 权限最小化：算法团队只读（浏览器看 dashboard），不发 kubectl 权限。
📝 算法指南📝 MLDiagnostics📝 架构师指南
5

性能优化：把瓶颈从「显存」转移到「通信」再逐个击破

这是方法论的核心。规模扩大后，瓶颈从单卡显存转移到跨卡通信。优化遵循「先测量、再针对带宽下手」的 roofline 思路。

诊断64→1024/2048 芯片扩展，forward 多 +8ms（其中 all-gather 6ms），backward 多 +23ms（其中 all-reduce 20ms）。根因：FSDP 的 all-gather 在 2D 平面只用到 3D torus 600GB/s 中的 400GB/s。
2D-FSDP用 fsdp + fsdp_transpose 双轴吃满 600GB/s，all-gather 从 5.58ms → 3.26ms（提速 38%，ICI 利用率 96%）。纯 DP 扩展只有 94% 天花板。
Cross-DCN DP把数据并行放到 DCN 独立信道，零 ICI 争用 → 扩展效率 99.2%。代价：mesh 拓扑配错会 -30%。
AOT 预编译本地提前编译，减少 TPU 空转和启动等待。
Remat贪心 13 轮搜索，找出 6 个张量的最优重计算组合，平衡显存与重算开销。
MoE kernelsindexed_gmm（-18GB）、gate+up GMM 融合、token permutation + tile、SparseCore collective offload、DataLoader prefetch（隐藏 470ms/step）、GA 权重提升修复（-30GB）。
📄 Scaling 理论分析📄 Cross-DCN📄 CP 理论📄 AOT
6

规模化量产：把「跑得快」变成「跑得稳」

到 2048 芯片量产规模，硬件故障成为主要敌人。需要一套可观测 + 隔离 + 复盘的运维体系。

  • 故障隔离方法论：2048 芯片 MoE 训练间歇性挂起（首 step 记指标时阻塞在 device→host 传输）→ 缩规模到 64 芯片逐节点池二分隔离 → 定位到单个故障节点池 tpu-64chips-prod-1 → 隔离替换。
  • 可观测关键指标：node_pool_status（RECONCILING>0 = 有 auto-repair）、interruption_count、HBM 带宽利用率、tpu_hlo_queue_size（管线拥塞）。
  • 组织保障：值班表轮转（5.23 起）+ 24×7 升级链（P0 直达 cloud-tpu-oncall，5 分钟响应）+ 线上问题复盘文化。

④ 启发 · 从这个项目里学到的

🎯
瓶颈是会迁移的

单卡时瓶颈是显存（linear attention 激活爆炸），规模化后瓶颈变成通信（all-gather/all-reduce）。优化的第一步永远是「重新测量当前瓶颈在哪」，而不是套用上次的结论。

🧭
带宽要看物理拓扑

3D torus 有 600GB/s，但朴素 2D-FSDP 只用 400GB/s——性能问题常常藏在「逻辑并行策略」和「物理网络拓扑」的错配里。2D-FSDP 和 Cross-DCN 本质都是「让通信模式贴合硬件拓扑」。

📐
精度对齐是不可跳过的地基

性能优化再漂亮，数值对不齐就是白做。逐算子对到 1e-4 看似笨重，却是后面一切实验可信的前提。

🔁
平台决定迭代速度

GitOps「改 3 行 YAML 即部署」+ 完善监控，让团队能高频试错。优化的产出 ∝ 迭代次数，而迭代次数 ∝ 平台顺手程度。

🔬
大规模故障靠「缩小复现」

2048 芯片的问题，靠在 64 芯片上二分逐池隔离定位。规模化运维的核心能力，是把大规模问题缩小到可复现的最小规模。

📌
把环境钉死

libtpu 0.0.36 这样的版本钉死，是无数次踩坑换来的。可复现性的代价，是对每个外部依赖版本的严格控制。
