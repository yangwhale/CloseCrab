# Review Request Prompt Templates

发给 reviewer（Bunny / Jarvis / Gemini UI Reviewer）的标准化 prompt 模板。三轮迭代各有侧重。

---

## 🎯 模板使用原则

1. **明确审查者身份**：告诉 reviewer 它是工程视角还是审美视角
2. **结构化提问**：列出 7 个维度，避免 reviewer 漏审
3. **要求 P0/P1/P2 优先级**：让你能批量决定先修什么
4. **附上文档链接 + 本地文件路径**：reviewer 能选对方式访问
5. **说明这是哪一轮**：R1/R2/R3 决定审查深度

---

## 📋 通用 Prompt 模板（适用所有轮次）

```
[Round X] Review 请求：{文档标题}

文档链接：{CC Pages URL}
本地文件：{绝对路径}

请从以下 7 个维度审查并提出具体改进意见：

1. **内容准确性**
   - 数据是否来源可靠 / 标注完整
   - 性能数字是否符合公开 benchmark
   - 模型 / 框架 / 版本号是否准确

2. **内容完整性**
   - 关键信息是否缺失（如 ETA、限制条件、前置依赖）
   - 数据矩阵是否覆盖主要场景
   - 决策路径是否覆盖典型用户

3. **结构与可读性**
   - 章节顺序是否符合阅读习惯
   - section 之间过渡是否自然
   - 重点信息是否突出（insight 卡片 / 加粗）

4. **视觉设计**
   - 风格是否符合 page-style 偏好（Material Design）
   - 配色是否克制（≤ 5 种主色）
   - 留白节奏 / 字号层级是否舒适

5. **图表质量**
   - SVG viewBox 是否自适应
   - 三种视觉语言是否独立运作
   - 文字密度 / 距框边 / 字号是否合理
   - 图例是否完整

6. **文案质量**
   - 中性度（不出现"标准答案"等绝对判断）
   - 营销腔避免（不出现"颠覆性""革命性"）
   - 单位 / 术语一致性
   - 时效性（数据有日期、Roadmap 有 ETA）

7. **缺失的图表**
   - 是否需要补：架构图 / 数据流图 / 决策树 / 时序图
   - 是否有内容更适合用图表表达

请打开链接查看实际渲染效果，然后给出结构化的 review 意见。
每条意见标注优先级：
- P0 = 必须改（结构破坏 / 数据矛盾 / 严重错误）
- P1 = 应该改（语义混淆 / 中性度问题 / 关键缺失）
- P2 = 建议改（polish / 可选增强）
```

## 🔄 三轮迭代专用模板

### Round 1：结构问题挖掘（重点 P0）

```
[Round 1] Review 请求：{文档标题}

文档链接：{URL}
本地文件：{路径}
当前完成度：约 70-80%

这是首轮 review，重点抓**结构性问题**：
- HTML 标签是否配对完整（建议先跑 grep 验证）
- 数据 / 决策树 / 文案之间是否有矛盾
- 关键 section 是否有截断
- 主要图表是否能正常渲染

不需要纠结细节文案。先把"骨架对不对"过一遍。

请按 7 个维度（准确性 / 完整性 / 结构 / 视觉 / 图表 / 文案 / 缺失）给意见，
重点列出 P0（必须改）问题。

预期产出：5-10 个 P0/P1，2-3 个 P2 提示。
```

### Round 2：细节问题打磨（重点 P1）

```
[Round 2] Review 请求：{文档标题}

文档链接：{URL}
本地文件：{路径}
当前完成度：约 90-95%

已根据 Round 1 修复了：{列出修了哪些}

这是第二轮 review，重点抓**细节问题**：
- 文案中性度 / 表述准确性
- 表格语义清晰度（如 ❓ 是 "未查到" 还是 "不支持"）
- 数据来源 / 时间标注是否完整
- 视觉细节（间距 / 对齐 / 颜色一致性）

请同时检查：
- Round 1 的修复是否到位
- 是否有新引入的问题（修一处坏一处常见）

预期产出：3-6 个 P1，2-4 个 P2，0-1 个 P0（如果有就严重了）。
```

### Round 3：增强内容 + 最终确认（重点 P2 + 总评）

```
[Round 3] Review 请求：{文档标题}

文档链接：{URL}
本地文件：{路径}
当前完成度：约 99%

已根据 Round 2 修复了：{列出}
新增内容：{列出新加的图 / 章节 / 数据}

这是最终确认轮，请判断：
1. P0 修复是否到位（有没有遗漏）
2. 是否有新引入的问题
3. Round 1/2 的 P2 建议是否仍然重要
4. 新增内容质量如何
5. 总评分更新（视觉 / 内容 / 结构 / 数据 / 完成度）
6. 是否可以发布？还是需要再迭代？

预期产出：发布 GO/NO-GO 决策 + 总评分。
```

## 🎨 Gemini UI Reviewer 专用模板

Gemini 在前端审美上更强，专门跑视觉 review。

```
让 Gemini 看看 UI：{文档标题}

文档链接：{URL}

请从前端审美视角 review，重点关注：

1. **视觉层级**
   - 标题字号比例是否合适（推荐 1.25 / 1.5 / 2 黄金比）
   - 行高是否舒适（正文 1.6-1.8）
   - 段落间距是否产生呼吸感

2. **配色和谐**
   - 主色 / 辅助色对比度是否符合 WCAG AA
   - 是否过度依赖某一种颜色
   - 状态色（success/warning/danger）是否一致

3. **留白节奏**
   - section 之间留白是否充足（推荐 60-80px）
   - 卡片内 padding 是否充足（推荐 24-32px）
   - 列表项间距是否舒适

4. **细节品质**
   - 圆角统一性（卡片 8-12px，按钮 4-6px）
   - 阴影层次（不要超过 3 种）
   - 边框使用克制（1px 即可，不要叠加多重边框）

5. **整体气质**
   - 是否有"AI 生成味"（紫色 / 渐变文字 / 霓虹光晕）
   - 是否符合 Material Design / Apple HIG / Stripe 等行业标杆
   - 给出 1-2 个最快能提升整体质感的修改建议

不需要看内容，专注视觉层。
```

## 🤝 多视角并行 Review 模板

让 Bunny + Jarvis + Gemini 并行各自跑 review，最后汇总对比。

### 给 Bunny 的（工程正确性视角）
```
请专注审查：
- 数据准确性
- 逻辑一致性（决策树 vs 表格不能矛盾）
- HTML 结构完整
- 中性度（不要营销腔）
不需要审视觉。其他视角有 reviewer 负责。
```

### 给 Jarvis 的（用户视角）
```
请站在"第一次接触这个对比的工程师"视角审查：
- 我能在 30 秒内理解 X 和 Y 的核心区别吗？
- 我能找到自己场景对应的推荐选项吗？
- 我会被哪个段落 confused？
- 哪个图表帮我理解最多？哪个最没用？
重点是"读完是否赋能决策"，不是技术细节。
```

### 汇总
三份 review 出来后，按维度合并：
- 三人都提的 → P0
- 两人都提的 → P1
- 单人提的 → P2（除非该 reviewer 在该维度是权威）

---

## 📊 P0/P1/P2 优先级判断标准

### P0（必须改，阻塞发布）
- HTML 结构破坏（标签未闭合、内容截断）
- 数据自相矛盾（决策树 vs 数据表）
- 关键事实错误（错误版本号、错误指标）
- 严重风格违规（违反 Chris 明确偏好如 page-style）

### P1（应该改，影响质量）
- 文案中性度问题（绝对判断、营销腔）
- 语义混淆（"未列出" 不区分 "没查" vs "不支持"）
- 数据缺时间 / 来源
- 视觉一致性问题（字号 / 颜色 / 间距不统一）
- 关键章节缺失图表

### P2（建议改，polish）
- 卡片信息密度过高
- 图表可以更精美
- 增加可选的对比维度
- ETA 可以更精确

---

## 🔁 Review 循环效率提升

1. **批量发请求**：一次列完 7 个维度，避免来回 ping-pong
2. **附完整上下文**：链接 + 本地路径都给，让 reviewer 选合适方式
3. **告诉哪一轮**：reviewer 知道 R1 vs R3 的预期不同
4. **承认前轮问题**：R2 列出 R1 修了什么，让 reviewer 知道哪些不用再提
5. **给"修好了再 review" 时间窗**：不要每个 commit 都求 review，攒一批
6. **存模板**：把这文件作为标准 prompt 存档，每次 copy-paste 改具体参数

## 💡 Bunny 特别说明

如果 reviewer 是 Bunny（CloseCrab bot），通过 Firestore Inbox 发请求：

```bash
python3 ~/CloseCrab/scripts/inbox-send.py bunny "$(cat <<'EOF'
[Round 1] Review 请求：TPU 推理框架对比

文档链接：https://cc.higcp.com/pages/tpu-frameworks-20260515.html
本地文件：/home/chrisya/.openclaw/canvas/documents/tpu-frameworks/index.html

请从 7 个维度审查...（粘贴上面的通用模板）
EOF
)"
```

Bunny 收到后会：
1. 用 Read 读本地 HTML（避免 IAP 重定向）
2. 按维度审查并标 P0/P1/P2
3. 通过 inbox 回复结构化 review

平均 R1 review 用时：5-10 分钟（取决于文档大小）。
